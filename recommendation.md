For production deployment, I would first split the current process into three concrete services on the same logical pipeline: a scheduler/frontier service, fetch workers, and a query/index service. The frontier should move from SQLite-backed in-process scheduling to Redis or RabbitMQ so worker capacity can scale without overloading one Python process. Per-host politeness should be enforced with a host-keyed token bucket, bounded retries, exponential backoff, and circuit-breaker behavior for unstable domains. Observability should also become first-class: Prometheus metrics for queue depth, fetch latency, status-code mix, and crawl throughput, plus Grafana dashboards and alerting for stalled queues or repeated host failures.

For search, I would replace the SQLite postings tables with OpenSearch or Elasticsearch and keep page parsing/index enrichment asynchronous. That would allow incremental document updates, richer ranking signals, and query isolation from crawl throughput. The resulting production stack would look like this: API layer in Python, Redis/RabbitMQ for the frontier, worker pools for fetching and parsing, OpenSearch for query serving, and Prometheus/Grafana for metrics. This keeps the current homework architecture recognizable while upgrading the exact components that become bottlenecks in a real deployment.
