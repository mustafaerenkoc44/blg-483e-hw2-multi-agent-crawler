For production deployment, the first step would be to separate the current single-process architecture into clearer fetch, index, and query responsibilities. The homework version uses SQLite and in-process worker threads because that is the right tradeoff for a localhost, single-machine assignment, but a production crawler would benefit from a durable work queue, stronger host-level politeness controls, structured observability, and a search tier optimized for heavier read traffic. Even before moving to multiple machines, per-host crawl budgets, retry policies, metrics, and alerting should be added so the system behaves predictably under failure and high-load conditions.

The second step would be to replace the local SQLite-backed search layer with infrastructure designed for larger indexing and query volume, such as OpenSearch, Elasticsearch, or a custom postings service. That transition would make it possible to add better ranking signals, incremental recrawling policies, richer content extraction, and safer operational controls without overloading the crawler process itself. In other words, the current design is a strong single-machine prototype, but productionization should prioritize isolation of responsibilities, crawl policy enforcement, and a more scalable search backend.
